## 概要

Codexで開発するときは、毎回の依頼に細かい前提を書くより、反復して守りたい開発ルールを `AGENTS.md` に寄せると安定します。

特に「どこを参照するか」「どこまで検証するか」「迷ったときにどう止まるか」を明文化すると、Codexの推測によるズレを減らせます。

## 基本方針

開発ルールは、Codexに「コードを書くだけでなく、変更の前提確認、テスト、レビュー、関連ドキュメント更新まで含めて作業する」ことを伝えるために使います。

公式ベストプラクティスでも、変更だけで終わらせず、必要なテストの作成・更新、関連チェックの実行、差分レビューまで含めることが推奨されています。`AGENTS.md` には、プロジェクトごとに繰り返し守るべき完了条件を短く固定します。

ただし、すべてを絶対条件にすると作業が止まりやすくなります。Dockerでの確認は「原則」、UI差分確認は「可能であれば」のように、環境差が出る項目は余白を残します。認証、課金、データ削除などの高リスク領域だけは、推測で進めず影響範囲を確認してから変更します。

## AGENTS.md に入れるルール例

```markdown
## Development Rules

- 要件や仕様は `docs/` フォルダ内のドキュメントを優先して参照する。
- 変更前に関連ファイルを確認し、既存の実装方針、命名規則、ディレクトリ構成、UI コンポーネントの使い方に従う。
- 不要な大規模リファクタリングは避け、依頼された目的に必要な範囲で変更する。
- タスクが大きい場合は、作業を小さな単位に分割し、依存関係と並列実行できる範囲を整理してから進める。
- コードを更新した場合は、必要に応じて関連ドキュメントとテストも更新する。
- コードの意図が分かりにくい箇所にはコメントを追加・更新する。ただし、明らかな処理に不要なコメントは追加しない。
- 動作確認やテストは、原則として Docker コンテナ内で実行し、ホスト環境への依存を避ける。
- デザイン実装・修正では、近接・整列・反復・コントラストを意識し、既存デザインとの一貫性を優先する。
- デザイン更新が入った場合は、関連する画面イメージやスクリーンショットを更新し、古い画像はアーカイブへ退避する。
- UI 変更時は、可能であれば変更前後の差分が分かる形で確認する。
- 仕様が不明な場合は推測で進めず、確認事項として明示する。
- セキュリティ、認証、課金、データ削除に関わる変更は、影響範囲を特に慎重に確認する。
- テストやビルドが失敗した場合は、失敗内容、原因、対応内容を簡潔に報告する。
```

## 使いどころ

このルールは、Webアプリや業務アプリのように、コード、テスト、仕様書、UI資料を一緒に保守するプロジェクトに向いています。

一方で、ドキュメントだけのリポジトリでは「Dockerでテストする」「UI差分を確認する」のような項目は不要な場合があります。`AGENTS.md` に入れる前に、そのリポジトリで実際に繰り返す作業だけに絞ります。

## 判断基準

開発ルールに入れるか迷ったら、次の基準で判断します。

- 同じ指示を2回以上Codexに伝えたものは、`AGENTS.md` に固定する。
- プロジェクトごとに違う確認コマンドや保存先は、具体的なパスやコマンド名まで書く。
- 環境によって実行可否が変わるものは、「原則」「可能であれば」と書き、確認できなかった場合の報告を求める。
- セキュリティ、認証、課金、データ削除のように失敗時の影響が大きいものは、確認事項として止まる条件を明記する。
- 一時的な方針や未確定の作業メモは、`AGENTS.md` ではなく `PLAN.md` や対象機能の仕様書に残す。

## 手順

1. まず `docs/`、`README.md`、既存コードを確認し、プロジェクトの前提を把握する。
2. 変更対象を小さく分け、依存関係がある作業と並列化できる作業を分ける。
3. 実装、ドキュメント、テストを同じ変更単位で更新する。
4. 変更に応じた確認コマンドを実行する。Docker構成がある場合は、原則としてコンテナ内で実行する。
5. 失敗や未確認事項があれば、原因、対応内容、残った確認事項を短く残す。

## 具体例

認証画面のUIを変更する場合は、先に `docs/specs/` や `docs/guidelines/` の要件を確認します。既存コンポーネントの命名や余白ルールに合わせて実装し、必要ならスクリーンショットを更新します。テストが失敗した場合は、失敗したテスト名、原因、修正した内容、まだ確認できていない点を完了報告に含めます。
