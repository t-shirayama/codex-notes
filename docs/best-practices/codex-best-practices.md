# OpenAI Codex ベストプラクティス

> 出典: [best-practices](https://developers.openai.com/codex/learn/best-practices)

Codex は「一回使い捨てのツール」ではなく、育てていくチームメンバーとして扱うのが効果的。

---

## 1. プロンプトの書き方

タスクを依頼するときは以下の4点を含めると精度が上がる。

- **目標:** 何を変えようとしているのか、何を構築しようとしているのか？
- **コンテキスト:** 関連するファイル・フォルダ・ドキュメント・エラーはどれか？（`@`でファイルをメンション可）
- **制約:** 従うべきアーキテクチャ・規約・安全要件は何か？
- **完了条件:** タスク完了前に満たすべき条件（テスト合格・動作変更・バグ修正など）は何か？

タスクの難易度に応じて推論レベルを調整する。

- **Low:** 範囲が明確な速いタスク
- **Medium / High:** 複雑な変更・デバッグ
- **Extra High:** 長時間・多段階・推論重視のタスク

---

## 2. 難しいタスクは計画から

複雑・曖昧・説明しにくいタスクは、コーディング前に計画を立てさせる。

- **Plan モード (`/plan` または `Shift+Tab`):** Codex がコンテキストを収集し、質問して、強固なプランを構築する。
- **インタビュー依頼:** 大まかなイメージしかない場合、先に Codex に質問させて曖昧さを解消する。
- **PLANS.md テンプレート:** 長時間・複数ステップの作業には実行プランテンプレートを用意する（[実行プランガイド](https://developers.openai.com/cookbook/articles/codex_exec_plans)参照）。

---

## 3. AGENTS.md でガイダンスを再利用する

プロンプトで毎回同じルールを書くのをやめ、`AGENTS.md` に移す。自動的にコンテキストへ読み込まれ、チーム全員が共有できる。

記載内容の例:

- リポジトリのレイアウトと重要なディレクトリ
- ビルド・テスト・Lint のコマンド
- エンジニアリング規約と PR 規則
- 制約事項と禁止事項
- 「完了」の定義と検証方法

**運用のコツ:**

- 短く正確に保つ。曖昧なルールが羅列された長いファイルは使いにくい。
- ファイルが肥大化したら、メインは簡潔にして計画・レビュー・アーキテクチャは別ファイルで参照する。
- Codex が同じ間違いを 2 回した場合、振り返りを求めて `AGENTS.md` を更新する。

---

## 4. 設定ファイルで一貫性を保つ

- 個人のデフォルト → `~/.codex/config.toml`
- リポジトリ固有の設定 → `.codex/config.toml`
- 一時的な上書き → CLI のコマンドラインオプション

新しい場合はデフォルトのパーミッションから始め、信頼できるリポジトリや特定のワークフローで必要になってから権限を広げる。

---

## 5. テストとレビューで信頼性を上げる

コードを書かせるだけで終わらせない。以下もセットで依頼する。

- 変更に対するテストの作成・更新
- 適切なテストスイートの実行
- Lint・フォーマット・型チェック
- 最終的な動作が要求と一致しているか確認
- 差分のバグ・リグレッション・危険なパターンのレビュー

`/review` コマンドでコードレビューを依頼できる（ベースブランチ比較・未コミット変更・コミット単位など）。`code_review.md` を作成して `AGENTS.md` から参照すると、レビュー動作を一貫させられる。

---

## 6. MCP で外部コンテキストを取得する

必要なコンテキストがリポジトリ外にある場合、MCP（Model Context Protocol）で外部ツールやシステムに接続する。プロンプトへのコピー＆ペーストが不要になる。

**使うべき場面:**

- 必要なデータがリポジトリ外にある
- データが頻繁に変わる
- ツールを繰り返し使う統合が必要

まず実際に手動ループを削減できる 1〜2 個のツールから始め、徐々に拡張する。

---

## 7. 繰り返し作業をスキル化する

同じプロンプトを何度も使ったり、同じワークフローを毎回修正しているなら、スキルにする時期。

**スキルの作り方:**

- 1 スキル = 1 業務に限定する
- 2〜3 個の具体的な使用例から始める
- 入力・出力・トリガーフレーズを明記する
- 最初からすべての例外を網羅しない。まず代表的なタスクで動かし、そこから改善する

**よいスキルの候補:**

- ログトリアージ
- リリースノート作成
- チェックリストに基づいた PR レビュー
- テレメトリ・インシデント要約
- 標準的なデバッグフロー

個人スキルは `$HOME/.agents/skills`、チーム共有スキルはリポジトリの `.agents/skills` に配置する。

---

## 8. 安定したワークフローは自動化する

ワークフローが手動で安定したら、Automations 機能でスケジュール実行する。不安定なうちに自動化しない。

**よい候補:**

- 最近のコミットの要約
- バグスキャン
- リリースノート作成
- CI 失敗の確認
- スタンドアップ要約

---

## 9. セッション管理

- **1 プロジェクト = 1 スレッドにしない。1 タスク = 1 スレッドが正解。** コンテキストが肥大化すると結果が悪化する。
- 作業が同じ問題の一部なら同じスレッドを維持する。真に分岐する場合のみ `/fork` する。

**便利なスラッシュコマンド:**

| コマンド   | 用途                                     |
| ---------- | ---------------------------------------- |
| `/plan`    | 計画モードに切り替え                     |
| `/resume`  | 保存した会話を再開                       |
| `/fork`    | 元のスレッドを保持しつつ新スレッドを作成 |
| `/compact` | 長いスレッドの以前の文脈を要約           |
| `/review`  | コードレビューを依頼                     |
| `/status`  | 現在のセッション状態を確認               |

---

## よくある失敗

- 毎回プロンプトに同じルールを書く → `AGENTS.md` またはスキルに移す
- ビルド・テストコマンドの情報を与えず、Codex に作業確認させない
- 複雑なタスクをいきなり実行させる → 計画フェーズを挟む
- ワークフローを理解する前に完全なアクセス権限を与える
- Git ワークツリーを使わず同じファイルで複数スレッドを並走させる
- 手動で安定していない段階で自動化する
- Codex を並行して使わず、一手ずつ見守る
