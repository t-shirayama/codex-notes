# AI駆動開発ワークフロー概要

> 出典: [CyberAgent Developers Blog](https://developers.cyberagent.co.jp/blog/archives/62010/)

OpenAIのCodexに代表される「大規模なコードコンテキストを理解するAIモデル」のポテンシャルを最大限に引き出すため、
各フェーズで使用するプロンプト、AIの予想出力、そしてコンテキスト管理の方法を体系化したワークフローの提案。

## AI駆動開発の核心原則

人間が「何を実現したいか(Why/What)」を定義し、
AIが「どう進めるか(How)」の最適な選択肢を提示する

## AI駆動開発のフェーズ全体

Phase0: プロジェクト構想: ビジョン策定
Phase1: プロジェクト初期化: フォルダ構造・AGENTS.md
Phase2: 要件定義: ユーザーストーリー・ドメインモデル
Phase3: アーキテクチャ設計: 技術スタック選定
Phase4: 環境セットアップ: パッケージ・ルール定義
Phase5: 機能実装: context.json管理

## Phase0: プロジェクト構想

コードを書く前に「なぜ作るのか」「誰のために作るのか」を明確にするフェーズです。
AIにビジネス観点の質問に答えてもらい、プロジェクトの方向性を固めます。
技術的な議論はこの段階では行いません。

## Phase1: プロジェクト初期化

プロジェクトの骨格を作るフェーズです。
AI駆動開発に最適化されたフォルダ構造と、AIが参照するメモリファイル（AGENTS.md）を生成します。
ここで定義した構造が、後続フェーズでの自動生成の基盤となります。

AGENTS.mdはAIの「長期記憶」として機能します。
プロジェクトルール、参照すべきドキュメント、現在のステータスを記載することで、AIが一貫した判断を下せるようになります。

## Phase2: 要件定義

「何を作るか」を詳細に定義するフェーズです。
ユーザーストーリー、UIワイヤーフレーム、ドメインモデルの3つを作成し、実装の土台を固めます。
技術に依存しない設計を心がけることで、後から技術選定を変更しても影響を最小限に抑えられます。

### ユーザーストーリー自動生成

人間は「ビジョンを読んでストーリーを生成して」と伝えるだけ。
AIがビジョンを分析し、必要なストーリーを特定して、自動的にフォルダ分けとファイル生成を行います。

### UIワイヤーフレーム生成

UIの構成をAIに生成させる際は、アスキーアートとMermaid図を組み合わせると効果的です。
成果物パスは指定せず、AIがルールに従って自動配置します。

### ドメインモデル設計

ドメインモデルは「ビジネスロジックの設計図」です。
RDBやNoSQLといった具体的な技術に依存せず、エンティティ、値オブジェクト、集約といった概念レベルで設計することで、技術選定の自由度を保ちます。

## Phase3: アーキテクチャ設計

小規模Webアプリでは、複雑なアーキテクチャは過剰になりがちです。
ドメインモデルを歪めない範囲で、最もシンプルな構成を選ぶことが重要です。

### 技術スタック選定

技術選定では「なぜその技術を選んだか」を比較表形式で整理します。
コスト、学習曲線、エコシステムの成熟度などを考慮し、プロジェクトの制約に最も適した選択を行います。

## Phase4: 環境セットアップ

Phase 3で選定した技術スタックを実際にインストールし、開発を始められる状態にするフェーズです。
人間は「インストールして」と伝えるだけで、AIがtech-stack.mdを読み取り、必要なパッケージと設定を自動で構築します。

### 技術スタックインストール

AIはtech-stack.mdを参照し、必要なパッケージをpackage.jsonに追加し、設定ファイルを生成します。

### プロジェクトルール・ガイドライン生成

project-rules.mdは「AIへの具体的な指示書」です。
命名規則、ディレクトリ構造、Git運用ルールを明文化することで、AIが一貫したコードを生成できるようになります。
このファイルが充実しているほど、Phase 5での自動生成の精度が向上します。

## Phase5: 機能実装とcontext.json管理

いよいよコードを書くフェーズです。
人間は「要件」だけを伝え、AIがAGENTS.mdとproject-rules.mdを参照して、適切な場所に適切な命名でファイルを自動生成します。
context.jsonで進捗と依存関係を管理することで、複数ストーリーの並行開発も可能になります。

### context.json の役割と構造

context.jsonは各ストーリーの「実装履歴」です。
どのファイルを作成・変更したか、どのストーリーに依存しているかを記録することで、AIが過去の実装を正確に把握できます。

### 機能実装

成果物のパスを羅列せず、要件と参照ドキュメントのみ伝えます。
AIがルールに従ってファイルを自動生成し、context.jsonで追跡します。
これが「要件ベースプロンプト」の核心であり、プロンプトの簡潔さと出力の一貫性を両立させます。

### context.json 更新プロンプト（実装完了時）

実装が完了したら必ずcontext.jsonを更新します。
ステータスを「completed」に変更し、作成・変更したファイルを記録することで、次のストーリー実装時にAIが正確なコンテキストを把握できます。

## フォルダ構成のサンプル

```bash
feature-request-board/
├── docs/
│   ├── specs/                  # 【SSOT】仕様書
│   │   ├── domain-model.md     # ドメインモデル定義
│   │   ├── api-specs.md        # API仕様
│   │   ├── db-schema.md        # DBスキーマ
│   │   ├── glossary.md         # 用語集
│   │   ├── project-rules.md    # プロジェクトルール
│   │   └── adrs/               # アーキテクチャ決定記録
│   │
│   ├── stories/                # ストーリー単位の作業
│   │   ├── story-000-setup/    # 初期セットアップ
│   │   │   └── vision.md
│   │   └── story-XXX-name/     # 各機能ストーリー
│   │       ├── requirement.md
│   │       ├── plan.md
│   │       ├── design.md
│   │       ├── wireframes.md   # UIワイヤーフレーム
│   │       └── context.json    # コンテキスト管理
│   │
│   └── guidelines/             # 開発ガイドライン
│       └── test-guidelines.md
│
├── src/                        # ソースコード
│   ├── app/                    # Next.js App Router
│   ├── components/             # UIコンポーネント
│   ├── lib/                    # ユーティリティ
│   └── types/                  # 型定義
│
├── tests/                      # テストコード
├── .env.example
├── README.md
└── AGENTS.md                   # AIアシスタント用メモリ
```
