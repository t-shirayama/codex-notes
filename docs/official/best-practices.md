# 公式ベストプラクティス要約

> - 種別: official
> - 参考元: [OpenAI Codex Best Practices](https://developers.openai.com/codex/learn/best-practices), [Review - Codex app](https://developers.openai.com/codex/app/review)
> - 最終ファクトチェック: 2026-05-04
> - 確認メモ: 参考元リンク、本文の主要論点、レビュー導線を確認

Codexは「その場しのぎの補助ツール」ではなく、ルールと手順を育てながら使うチームメンバーとして扱うと安定します。

## このページの見方

- 公式が何を勧めているか
- 実務ではどう効くか
- すぐ真似できる短い例

## 1. プロンプトの書き方

- **目標:** 何を変えたいか
- **コンテキスト:** 関連ファイルやエラーは何か
- **制約:** 守るべきルールは何か
- **完了条件:** 何を満たせば終わりか

### 実務での意味

依頼の粒度が揃うと、Codexが余計な推測をしにくくなります。

### 具体例

```txt
目的: CSVインポートのバグ修正
対象: @backend/importer.py
制約: 既存APIは変えない
完了条件: 失敗ケースのテストを追加する
```

## 2. 難しいタスクは計画から

- 複雑な作業は `/plan` を使う
- 曖昧な依頼は先に質問させる
- 長い作業は `PLANS.md` で決定事項を残す

### 実務での意味

要件が曖昧なまま実装に入るより、先に論点を洗い出した方が差し戻しが減ります。

## 3. AGENTS.md でガイダンスを再利用する

- 毎回同じルールを書くのをやめる
- 恒久ルールは `AGENTS.md` に移す
- ルールが肥大化したら別ファイルへ分ける

### 具体例

`変更前に意図を説明する`、`テストコマンドを明記する` のようなルールは、毎回のプロンプトではなく `AGENTS.md` に寄せると効果的です。

## 4. 設定ファイルで一貫性を保つ

- 個人の設定は `~/.codex/config.toml`
- リポジトリ設定は `.codex/config.toml`
- 最初から権限を広げすぎない

### 実務での意味

事故の余地を減らしながら、必要な範囲だけ自動化できます。

## 5. テストとレビューまで含めて依頼する

- テストの追加・更新
- 必要なチェックの実行
- 差分レビュー

コードを書かせるだけで終わらせず、検証まで含めて運用します。

### `/review` と diff レビュー

PR前、広範囲変更後、またはレビュー観点を固定したいときは、実装依頼とは別にレビュー依頼を出します。Codex app のレビュー画面では未コミット差分やブランチ差分を確認でき、`/review` を使うと未コミット変更、base branchとの差分、特定コミットなどをレビュー対象にできます。

### 具体例

```txt
/review
未コミット変更をレビューして。バグ、仕様 regressions、足りないテストを優先して指摘して。
```

実装とレビューを分けると、Codexに「作る視点」と「壊れていないか確認する視点」を切り替えさせやすくなります。チームでレビュー観点を揃えたい場合は、`code_review.md` のようなファイルを作り、`AGENTS.md` から参照します。

## 6. 外部コンテキストはMCPで取得する

- リポジトリ外の情報が必要
- 更新頻度が高い
- 手動コピペを減らしたい

## 7. 繰り返し作業をskill化する

- 1 skill = 1業務
- 具体例から始める
- 入出力と使いどころを明記する

### 具体例

- リリースノート作成
- ログ要約
- PRレビューの観点整理

## 8. 安定したワークフローは自動化する

手動で安定したあとに、自動化へ進めます。

## 9. 1タスク1スレッドで扱う

- 問題ごとにスレッドを分ける
- 本当に分岐したときだけ `/fork` する

## まとめ

- 毎回の指示より、再利用できるルールを増やす
- 難しい作業は計画を先に固める
- テスト、レビュー、設定まで含めて運用する
