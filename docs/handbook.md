# Codex運用ハンドブック

> - 最終確認: 2026-05-30
> - 位置づけ: 自分がCodexを日常運用するときの最小ルール

## Codexを使う基本姿勢

Codexには、作ってほしいものだけでなく、前提、制約、完了条件を渡します。コード生成だけで終えず、調査、計画、実装、確認、レビューまでを1つの流れとして扱うと安定します。

人間は「なぜやるか」「何を成功とするか」を決め、Codexには「どう調べ、どう変更し、どう確認するか」を任せます。曖昧なまま大きな作業を投げず、小さく切って進めます。

## 依頼の書き方

依頼はIssueのように書きます。最低限、目的、対象、制約、完了条件を分けます。

```txt
目的:
請求履歴画面で、失敗した決済だけを絞り込めるようにする。

対象:
src/pages/BillingHistory.tsx
src/api/billing.ts

制約:
既存のAPIレスポンス形式は変えない。
UIコンポーネントは既存のFilterSelectを使う。

完了条件:
失敗ステータスで絞り込める。
関連するテストを追加する。
変更差分をレビュー観点つきで要約する。
```

短い作業でも、対象と制約を入れるだけでズレが減ります。背景説明を長くするより、変更対象と終わり方を明確にします。

## 計画から始める判断基準

次の条件に当てはまる作業は、実装前に計画から始めます。

- 複数ファイルや複数レイヤーにまたがる
- 仕様が曖昧
- 既存設計との整合性が重要
- 失敗時の影響が大きい
- 認証、課金、権限、データ削除に触る

計画依頼では「まだ実装しないで、影響範囲、確認事項、実装手順、テスト観点だけ出して」と明示します。計画が固まったら、作業単位を小さくして実装へ進めます。

## 実装・確認・レビューの流れ

通常の作業は次の順で進めます。

1. 関連ファイルと既存ルールを確認する
2. 変更範囲を小さく決める
3. 実装する
4. 変更に応じたテストや確認を行う
5. 差分をレビュー観点で確認する
6. 未確認事項や失敗した確認を短く残す

Codexに実装を任せるときは、テストやレビューも同じ依頼に含めます。失敗した場合は、失敗内容、原因、対応内容、残った確認事項を報告させます。

## 開発プロジェクトで固定したいルール

この章は、自分の開発プロジェクトで `AGENTS.md` に移す候補です。すべてを一度に入れるのではなく、実際に繰り返す作業だけを選んで固定します。

### 変更前の確認

- 要件や仕様は、まずプロジェクトの `docs/` を参照する
- 変更前に関連ファイルを確認し、既存の実装方針、命名規則、ディレクトリ構成に従う
- 依頼目的に必要な範囲で変更し、不要な大規模リファクタリングは避ける
- タスクが大きい場合は、作業を小さな単位に分割し、依存関係と並列実行できる範囲を整理する

### 変更時の原則

- コードを更新したら、関連ドキュメントとテストも必要に応じて更新する
- コードだけでは処理の意図が分かりにくい箇所には、短いコメントを追加または更新する
- 明らかな処理には不要なコメントを追加しない
- 仕様が不明な場合は推測で進めず、TODO や確認事項として明示する

### UI / デザイン

- デザイン実装や修正では、近接、整列、反復、コントラストを意識する
- 既存のデザインシステムやコンポーネントの一貫性を優先する
- UI 変更時は、可能であれば変更前後の差分が分かる形で確認する
- デザイン更新が入った場合は、関連する画面イメージを差し替え、古い画像はアーカイブへ退避する

### 検証

- 動作確認やテストは、ホスト環境に依存しないように Docker コンテナ内で実行する
- テストが失敗した場合は、失敗内容、原因、対応内容を簡潔に報告する
- 確認できなかった項目は、未確認として残す

### 高リスク領域

セキュリティ、認証、課金、データ削除に関わる変更は、特に慎重に影響範囲を確認します。実装前に対象データ、権限、失敗時の影響、ロールバック方法を整理し、推測だけで進めません。

### CI / ブランチ運用

GitHub Actions の CI は、ひとまとめにせず段階を分けます。まず静的チェック、次に単体テスト、統合テスト、最後に e2e という順にすると、失敗箇所を追いやすくなります。

```txt
step1: backend-check / frontend-check
step2: backend-unit / frontend-unit
step3: backend-integration / frontend-integration
step4: e2e
```

ブランチは `main`、`develop`、`features/*` のように役割を分けます。`main` と `develop` は保護し、レビュー必須、テスト通過必須、直push禁止を基本にします。

### 言語設計

プロジェクト固有の概念は、ユビキタス言語として定義します。日本語の業務語と対応する英語を並べておくと、実装時の変数名、クラス名、API名、画面文言が揃いやすくなります。

## AGENTS.mdに残すべきルール

`AGENTS.md` は、毎回守ってほしい恒久ルールだけを置きます。タスクごとに変わる背景や長い説明は入れません。

残すとよいもの:

- リポジトリの目的
- 情報源の優先順位
- 編集対象と削除方針
- 参考元の残し方
- 完了時の確認方法
- 繰り返し起きる失敗への対策

同じ指示を2回以上書いたら `AGENTS.md` 候補です。ただし、具体的な手順が長くなるなら `docs/handbook.md` に寄せます。開発プロジェクトでは、上の「開発プロジェクトで固定したいルール」から、そのリポジトリで本当に使うものだけを選んで移します。

## skills / hooks / MCP を使う判断基準

使い分けは次のようにします。

| 仕組み | 使う場面 | このリポジトリでの方針 |
| --- | --- | --- |
| AGENTS.md | 毎回守る恒久ルール | ルートに最小限だけ置く |
| skill | 会話から呼び出したい再利用手順 | 実体は同梱せず、必要なら個人環境で作る |
| hook | 実行前後の定型チェック | スクリプトや実設定は同梱しない |
| MCP | 外部ツールや最新情報をCodexに渡す | 手順だけ残し、実設定は個人環境に置く |
| plugin | 複数のskillや連携設定を配布する | このリポジトリでは扱わない |

skillは、入力、出力、判断基準が固定できる作業に向いています。hookは、危険操作の確認や定型通知のように機械的に走らせたい処理に向いています。MCPは、公式ドキュメント、Issue、ログ、外部サービスの状態など、リポジトリ外の情報を扱うときに使います。

まだ手順が揺れている作業は、自動化せず文章化から始めます。

## セッション管理と並列作業

基本は1タスク1スレッドにします。同じ問題の続きなら同じスレッドを使い、別案を試すときだけ分岐します。

複数作業を並行する場合は、同じファイルを複数スレッドで触らせません。探索、調査、レビューのように境界がはっきりしている作業だけ分けます。

長い作業では、途中で状態、権限、差分、残り作業を確認します。CLIでは `/status`、`/diff`、`/review`、`/compact` のような節目のコマンドを使うと文脈を保ちやすくなります。

## よくある失敗

- 毎回のプロンプトに恒久ルールを詰め込む
- 対象ファイルや完了条件を書かない
- 複雑な作業なのに計画を省く
- テストや確認方法を依頼に含めない
- 権限を広げる理由を確認しない
- 同じファイルを複数スレッドで触らせる
- 手動で安定していない作業を急いで自動化する
- 旧記事を残しすぎて、どれを読めばよいか分からなくする
